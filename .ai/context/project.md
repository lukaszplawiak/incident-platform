# Project Context

## Project Name

Incident Platform

---

## Purpose

Incident Platform is a production-oriented incident management system
designed for IT operations teams, system administrators, and Security
Operations Center (SOC) teams.

The system helps organizations detect, manage, investigate, and resolve
operational and security-related incidents faster.

The primary goal is reducing the time between:

"an anomaly is detected"

and

"the responsible team starts mitigation."

---

## Problem Domain

Modern organizations operate complex systems consisting of:

- applications
- infrastructure
- databases
- cloud services
- security systems

Failures and anomalies can happen at any time.

Without a centralized incident management process:

- alerts become noisy,
- ownership is unclear,
- response is slower,
- investigation history is lost,
- lessons from previous incidents are not captured.

Incident Platform provides a structured workflow for handling incidents.

---

## Core Domain Concepts

The system revolves around these concepts:

### Incident

The central business entity representing a problem requiring investigation
and response.

### Alert

An external signal generated by monitoring systems.

Examples:

- high CPU usage
- unavailable service
- suspicious authentication activity

### Notification

Communication sent to responders.

Examples:

- email
- Slack
- SMS

### Escalation

Automatic escalation process when an incident is not acknowledged
or resolved within defined time limits.

### Audit Event

Immutable record of important system actions and state changes.

---

## Architecture Principles

Agents modifying this project must respect these principles:

### Domain first

Business rules belong in the domain layer.

Do not move business logic into:

- controllers
- repositories
- configuration classes

---

### Explicit design decisions

Prefer:

- simple solutions
- explicit code
- clear ownership

Avoid unnecessary abstraction.

---

### Security first

The system must preserve:

- tenant isolation
- authorization boundaries
- auditability
- secure authentication flows

---

## Current Development Phase

The project is currently implementing the classic
human-driven incident response model.

The system does not currently perform autonomous incident analysis
or automated remediation.

Future extensions may build on top of this foundation.
